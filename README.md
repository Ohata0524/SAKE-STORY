# SAKE STORY - 日本酒にストーリーを添えて、あなたに合った一本と出会う -

## コンセプト
**「スペックではなく、物語で選ぶ」**
SAKE STORYは、日本酒の基本情報だけでなく、そのお酒が生まれた背景や蔵元の想い（ストーリー）を軸に、日本酒に詳しくない人から愛好者まで、年齢問わず「自分に合う1本」や「贈り物として喜ばれる1本」と出会えるマッチングプラットフォームです。

### ターゲット層
* **20代〜80代**の幅広い層
* 自分用に新しい日本酒を開拓したい方
* 大切な人への失敗しない贈り物（ギフト）を探している方

## 開発の背景
**「デジタルの『ぽんしゅ館』を作りたい」**
私の地元・新潟には、多種多様な日本酒を気軽に試飲できる「ぽんしゅ館」という施設があります。そこで偶然出会った一杯に感動した経験が、このアプリの原点です。
物理的な距離に関わらず、オンライン上でも「自分の好みに合った予期せぬ一本」と出会える環境を作りたいと考え、開発に至りました。

## 課題と解決策
| 現状の課題 | SAKE STORYの解決策 |
| :--- | :--- |
| **種類が多すぎて専門知識がないと選べない** | 直感的な目的別クイックフィルタにより、膨大な選択肢から最適な一本を提示します。 |
| **ギフト用／自分用の選び分けが困難** | 利用シーンによる絞り込みと、ストーリー情報による「語れる付加価値（文脈）」を提供します。 |
| **レビューや評価の基準が分散している** | サイト内レビュー機能（星評価）により、実際の飲用者の声を一元管理し、信頼できる比較情報を提供します。 |
| **知名度の高い一部の銘柄に認知が偏る** | トップページにランダムシャッフル機能を実装し、まだ見ぬ優良銘柄との「一期一会の出会い」を演出します。 |

## 利用フロー
1. **年齢確認**: 初回アクセス時に年齢確認を実施（20歳未満の利用防止）。
2. **出会い・検索**: トップ画面の「ランダムおすすめ」や、「甘口/辛口」「ギフト用/自分用」などの条件から絞り込み。
3. **ストーリー体験**: 商品詳細ページにて、スペックだけでなく「誕生秘話」や「おすすめの飲み方」を閲覧。
4. **選定・購入**: 気に入った商品は、メーカー公式サイト等の購入ページへ遷移。
> ※アプリ内決済・在庫管理（EC機能）は非保持。公式サイトがない銘柄は「銘柄名 + 購入」でGoogle検索結果へ遷移させ、購入機会を損失させない工夫をしています。
5. **体験の共有**: 飲んだ感想をレビューとして投稿し、他のユーザーへ共有。

## 機能一覧
### 実装済み機能
* **認証機能**: ユーザー登録、ログイン、ログアウト
* **年齢確認**: 初回利用時のモーダルによる確認
* **トップ画面シャッフル**: アクセスごとにDB側で抽選を行う、おすすめ銘柄のランダム表示機能（認知の偏りを防ぐアルゴリズム）
* **商品検索・詳細フィルタ**: キーワード検索、詳細フィルタ（甘辛、産地、価格帯、利用シーン等）
* **詳細表示**: 基本情報（味、ペアリング、地域）＋ ストーリー（読み物として独立した画面設計）
* **レビュー機能**: 評価（★1〜5）とコメントの投稿・閲覧・削除
* **お気に入り機能**: 気いる銘柄の保存・削除
* **外部連携**: 蔵元/販売サイトへの遷移ボタン（公式サイトがない場合、Google検索結果ページへ遷移）

### 実装予定機能
* **レコメンド機能の強化**: ユーザーの閲覧履歴やお気に入りデータに基づく提案アルゴリズムの導入
* **最近見た商品リスト**: 比較検討をスムーズにするための閲覧履歴の保存・表示機能
* **SNSシェア機能**: お気に入りの銘柄を魅力的なカード形式でX（旧Twitter）やInstagramへシェアできる機能
* **多言語対応**: インバウンド需要を見据えた、英語等への切り替え機能

### やらないこと（非スコープ）
* **EC機能（決済・在庫管理）**: 外部遷移に閉じることで開発コストを削減し、「出会い」の提供に特化。
* **日本酒のボトル・ラベル写真の使用**: 酒蔵の著作権や肖像権の侵害リスクを回避し、法的安全性を担保。

## 技術的・デザイン的な取り組み

### 1. UI/UX・アクセシビリティへの配慮（Figma設計）
20代〜80代までの幅広いユーザー層が迷わず快適に利用できるよう、Figma上でコンポーネント化とデザインシステム構築を行いました。
* **タイポグラフィの階層化**: 高齢者層の視認性を考慮し、本文のベースサイズを16px（Regular）に固定。視覚的な情報階層を厳格に整理。
* **セマンティックカラー設計**: ストーリー表示エリア専用の背景色を設け、他のスペック情報から視覚的に独立させることで「読む体験」を強調。
* **Auto Layoutによる誤操作防止**: ボタン等のタップ領域を適切に確保し、誤操作を防ぐ設計をコンポーネント単位で実現。

### 2. 堅牢なデータバリデーション (Zod)
Supabaseから取得する外部データに対し、Zodを用いたランタイムバリデーションを導入。ドメイン層の入り口で型安全性を保証し、予期せぬデータ不備によるランタイムエラーを未然に防いでいます。

### 3. 関心の分離を意識したアーキテクチャ
UI、ドメインロジック、インフラストラクチャ（API連携）を明確に分離し、保守性の高いコードベースを構築。
* **Domain**: ビジネスロジックとバリデーション定義
* **Infrastructure**: Supabase等の外部サービス依存の集約
* **Hooks / Components**: 表示とUIロジックの担当

### 4. Atomic Designに基づくコンポーネント設計
コンポーネントを atoms, molecules, organisms に分類して再利用性を向上。ビジネスロジックはカスタムフックに抽出することで、UIとロジックを完全に分離しました。

### 5. 保守性を重視したCSS設計
Tailwind CSS v4を採用し、独自のカラーパレットや共通変数を管理。デザインの一貫性とコードの可読性を両立させています。

### 6. CI/CD基盤と品質管理 (GitHub Actions & Jest)
「壊れない開発」を実現するため、GitHub Actionsを利用したCIパイプラインを構築。PR作成時に自動でLint/BuildチェックおよびJestによる単体テストを実行し、すべてパスしなければマージできないフローを確立しています。

## 環境構築
ローカル環境で実行するための手順です。

```bash

# 2. ディレクトリへ移動
cd sake-story

# 3. パッケージのインストール
npm install

# 4. 環境変数の設定
# .env.localファイルを作成し、Supabaseのキーなどを設定してください。
# NEXT_PUBLIC_SUPABASE_URL=[https://your-project.supabase.co](https://your-project.supabase.co)
# NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

# 5. ローカルサーバーの起動
npm run dev
